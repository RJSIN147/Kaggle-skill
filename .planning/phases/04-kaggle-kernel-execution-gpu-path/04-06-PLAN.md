| kernel_log.txt → recorder | pulled Kaggle log text; already treated as untrusted (scanned, never echoed) — WR-03 extends this so an unreadable log fails closed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-06-01 | Tampering | record_experiment.py status rung | mitigate | Match `kernel_run.json.status` against the EXACT allowlist `{"ERROR","CANCEL_ACKNOWLEDGED"}` (membership test only); never substring/regex/eval; a crafted status string can only ever cause a FAILED classification, never force a SUCCESS |
| T-4-06-02 | Info-Disclosure | recorder output | mitigate | The status value and log contents are used for classification and written to meta.json only — NEVER printed to stdout/stderr (preserves truth 5, never-echo) |
| T-4-06-03 | DoS / Elevation | malformed kernel_run.json | mitigate | Read via the existing fail-clear `_read_json` (returns None, never raises); a missing/garbage file does not crash the recorder and does not trigger the fail rung, so it can never flip a failure into a SUCCESS |
| T-4-06-04 | Spoofing | stale result.json off a re-pushed slug | mitigate | The authoritative kernel_run.json.status rung + WR-03 unreadable-log rung both decide BEFORE `_validate_result`, so a stale valid result.json can never rescue a Kaggle-confirmed or unverifiable-log failure |
| T-4-06-SC | Tampering | package installs | mitigate | No package installs — stdlib-only edit to one script + tests; `uv run pytest` uses the already-declared dev deps (no runtime `pip install`) |
