<objective>
Build `check_submission.py` - the FREE entry point, and the whole point of D-14's three-way split.
It answers "should I submit this?" completely: it refuses the wrong competition type, validates the
file against the sample, counts the Kaggle-authoritative remaining budget, compares this experiment's
CV against the best already-submitted CV with a fold-noise bound, and renders the decision material -
and it NEVER spends a slot doing any of it.

The policy lives in a separate pure module (`submission_gate.py`) with zero I/O, so the D-05/D-06/D-08
matrix is exhaustively testable with no filesystem and no network.

D-05 is load-bearing and must be held exactly: the framework never auto-submits and never silently
hard-refuses. It computes and presents the material and TAKES A POSITION (BLOCKED by default when the
gain is within noise), but the human always makes the final call. This is PROJECT.md's
anti-"opaque budget-burning agent" principle made mechanical.
Output: `scripts/submission_gate.py`, `scripts/check_submission.py`.

    NOISE_K_DEFAULT = 1.0. State the reasoning in the module docstring: cv_std from run_cv is the
    POPULATION std of the fold scores, not the standard error of the mean (~2.2x smaller at 5 folds),
    so requiring the gain to exceed one full fold-std is a deliberately CONSERVATIVE bar. The right
    default when the protected resource is a scarce, irreversible daily slot: a false "submit" wastes
    a slot; a false "blocked" costs one keystroke, because D-05 guarantees the human can override.
    Configurable via config.json -> submission.noise_k for a high-variance competition.

    is_meaningful(cand_cv, cand_std, best_cv, greater_is_better, k=NOISE_K_DEFAULT) -> bool (D-06):
      - best_cv is None (no prior SUBMITTED experiment) => return True. The FIRST submission is CLEAR.
        An empty comparison set must NEVER produce a spurious block. State this explicitly.
      - margin = (cand_cv - best_cv) if greater_is_better else (best_cv - cand_cv)
      - return margin > k * cand_std   # STRICTLY greater; a gain exactly equal to the noise bound is
                                       # NOT meaningful. A gain in the wrong direction is negative and
                                       # therefore never passes.
      - cand_std missing/None/0 -> treat 0 as a zero noise bound (any positive margin passes) but
        guard against a non-numeric std by returning False (fail closed on garbage input).

    decide(*, cand_cv, cand_std, best_cv, greater_is_better, remaining, limit_provenance,
           k=NOISE_K_DEFAULT) -> dict (D-05 + D-08):
      Returns {"recommendation": "SUBMIT" | "BLOCKED", "reasons": [str, ...],
               "warnings": [str, ...], "requires_confirmation": bool}
      Rules, in order:
      1. remaining is None (the budget could not be established) => BLOCKED, requires_confirmation
         False, reason naming the fail-closed cause. Never guess a count (D-04).
      2. remaining <= 0 => BLOCKED regardless of the CV signal; reason names the exhausted budget.
      3. limit_provenance == "assumed_default" => append a warning to EVERY decision naming the budget
         as ASSUMED (5/day - not confirmed against the rules page) (D-08). AND when remaining == 1 (the
         LAST assumed slot), set requires_confirmation True EVEN IF the CV recommendation is SUBMIT -
         if the real limit is lower, that slot may not exist at all.
      4. CV signal: is_meaningful(...) True => recommendation SUBMIT, with a reason stating the margin
         and the noise bound (k * cv_std) in numbers. False => recommendation BLOCKED and
         requires_confirmation True - the human must CONSCIOUSLY confirm to proceed (D-05
         block-by-default with an informed override).
      Every reason and warning is a plain, human-readable string that names the actual numbers. The
      renderer prints them; the SKILL holds the human loop.

      D-07: the override reason is OPTIONAL. This module never demands one and never returns a state
      that requires one.

    ⚠ DO NOT build any final-selection / nomination advisory or recommendation (D-12: explicitly and
    deliberately scoped OUT of v1 by the user, despite roadmap 05-02 naming it). Not here, not
